Node

对于一个集群会包含几个节点，一个是 master 节点，另外还有一些 worker 节点，在 master 节点上会运行一些核心服务，比如调度，API服务等，在 worker 节点上主要完成一些任务的调度，比如部署一些容器或者 APP ，一般会选择一些特定的 node 也就是 worker 节点上进行部署

Pod

一个 Pod 中可能存在多个 Container,也就是说一个Pod中只少有一个 Container

如何将一组容器划分为一个 Pod 呢？它们之间必然有一些共同的特性

  1. 一个或者一组应用容器，它们分享物理资源，比如 volume

  2. 分享相同的命名空间的容器

比如网络空间,具有相同的IP地址，就类似与两个容器运行在一台机器上，如果他们之间要通信是没有限制的，因为在同一台机器上，就可以通过127.0.0.1，就相当于在一台电脑上启了两个进程，比如一个后台，一个数据库，它们之间是可以相互通信的

  1. Pod 是k8s中 最小 的调度单位

k8s 有个组件叫 scheduler,当给k8s下达命令创建pod时，scheduler会到k8s的节点上看应该将pod创建在哪个节点上，这就是调度，调度是以pod为单位的，并不是以容器为单位

同一个 pod 的容器肯定是在同一个节点上，不会被分到两个不同的节点上，因为pod已经是最小的了

Namespace

  • Namespace 命名空间用于不同的 team,不同 project 之间的隔离

比如有不同的项目或者不同的团队在使用同一套k8s环境最好使用 namespace 坐下隔离，假设用了同一个 namespace,创建了一个nginx的pod,另外一个人也创建了nginx的pod,就会出现名称冲突的情况，所以可以通过不同的命名空间将不同的team隔离开

  • 在不同的命名空间中，各种资源的名字是相互独立，比如可以具有相同名称的pod存在

Deployment

在k8s架构中分为 master 节点和 worker 节点，在 master 节点上会有一些服务，比如 etcd (分布式key-value存储的数据库), API Server(对外通信比如使用kubectl就可以连接API Server), Scheduler(当创建pod时会去调度，那个几点上资源比较充足，然后决定pod创建在哪个node),Controller Manager
