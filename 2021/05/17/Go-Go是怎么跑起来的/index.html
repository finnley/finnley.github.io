Go 程序 hell.go 的编译过程

  1. 文本
  1. 编译

单个文件可以通过 go build xxx.go 生成可执行文件，这就是简单的代码编译过程，从文本编译到二进制文件，其实这个编译过程也是有很多阶段的，这些属于编译原理内容（有时间可以研究）

可以通过 go build -x hello.go 命令在编译的时候可以将详细内容输出出来

编译：文本代码 -> 目标文件（.o, .a）
链接：将目标文件合并为可执行文件

编译阶段就是将文本代码转成目标文件，这个目标文件需要和标准库里已经编译出来的静态库做链接，链接了之后就会在临时目录里面把编译后完整的二进制文件输出出来，它再把临时目录中的文件挪动到默认编译出来 -o 的目标，然后再将临时目录删掉，这就完成了整个编译过程

  1. 解析 ELF Header

  2. 加载文件内容至内存

计算机是怎么执行我们的程序的呢？

目标：了解计算机CPU是怎么执行二进制文件中的指令

从文本代码编译之后，最终生成文件里面只有汇编指令，有一个非常常见的通用寄存机叫 pc 寄存器，CPU就是靠 pc 寄存器 所指向的位置来执行我们的代码的，在 64 位平台上 pc 寄存器 = rip

CPU 无法理解文本，只能执行一条一条的二进制机器码指令
就是一行一行解析后面的二进制代码然后再去执行的，每次执行完一条指令，pc 寄存器就指向下一条继续执行，对于CPU来说没有什么负担，就挨个去解释每条指令并执行就行了

runtime

目标：了解 Go 语言本身编译出来的二进制在预执行起来之后是怎么组成的

GO 的可执行文件由两部分组成

  1. runtime

  2. 用户代码

Go 语言是一门有 runtime 的语言，可以认为 runtime 是为了实现额外的功能，有程序员提供的在程序运行时自动加载/运行的一些模块

上面是我们些的 GO 程序代码，下面就是 Runtime ，GO 代码通过一些比如 goroutine 的创建，或者 channel 的操作，或者内存的分配来和 Runtime 进行交互，Runtime 负责管理我们提交的一些任务，它和操作系统是通过系统调用和线程的调用来做交互的，也有人说 runtime 相当于 go 的 kernel ，可以这么理解

  • Go 语言的 runtime 包括：
  1. Scheduler

调度器管理所有的 G，M，P，在后台执行调度循环

  1. Netpoll

为了支持高并发，原生的在 runtime 里面实现了一套网络 FD 管理流程

  1. Memory

当代码需要内存时，负责内存的分配工作

  1. Garbage

当内存不再需要时，负责内存回收

这些模块中（runtime 中的四座大山），最核心的就是 Scheduler 调度器，它负责串联所有的 runtime 流程

调度器从什么地方启动

通过 entry_point 找到 Go 进程的执行入口，它会通过三个汇编函数一路跳到 runtime.rt0_go 这个函数里面，这个函数是初始化的一个非常重要的函数，它会执行参数的初始化，全局 m0, g0 的初始化，这是一个特殊的线程，m0 执行的是应用的主线程；
获取CPU的核心数；
初始化内置的数据结构；
执行用户的main主函数，这些就是主线程要做的事情

m0: Go 程序启动后创建的第一个线程；

调度组件与调度循环

每次写下上面的一段语句的时候，到底发生了什么？

其实就是向 runtime 提交了一个计算任务。func() { xxx } 里面包裹的代码就是计算任务的内容，它包含的一坨代码，这坨代码已经被编译到二进制文件里面了，sync 包其实并不是 runtime 的一部分，sync 包只是 go 语言的一个标准库，它依赖了 runtime 提供的一些组件。

既然有提交任务，就会有人去消费这个任务，所以 Go 的调度流程本质上是一个 生产-消费 流程

平时开发中会写一些 生产-消费 程序，常见就是消息队列，在右边生产端会不停的提交任务，对于程序来说就是每次 go func 一次都是提交一个任务，这个任务肯定会进入某个队列，然后在队列的另一端会有人不停的消费这个任务去执行，执行完之后相当于这个任务被执行掉了，对于用户来说就是目标达到了，把我们的代码都执行了一遍

在 runtime 中，这些都是有一些角色的，右边的生产者相当于每次 go func() {} 的时候都生产了一把，中间的队列（queue），任务会被打包成 协程 ，协程 在上面的模型叫 G， 也就是 goroutine ，goroutine 它最终是进入了专门的队列，队列的另一端消费的就是 线程-M，线程是一直在循环执行消费操作来消费这些任务

为了能够在性能上达到更好的目标，GO语言在队列上是有分级的，中间的队列会分为几个部分，比如有一个本地的执行队列 local run queue ，还有一个全局的执行队列 global run queue，当然实际上不止这两个，对于 runtime 来说，它就是来协调这些对象之间的工作，并且进一步去做优化

go func 去了哪里？

右边黄色，蓝色，绿色部分就是我们的线程，大部分线程（M）都是一直在执行调度循环，调度循环简单理解就是从左边的各种队列里面把任务拿出来去执行，这就是线程的相关操作。

左边的三个蓝色的就是任务来源，队列有一个本地队列 local run queue 和一个全局队列 global run queue，为了优化，GO 语言还专门做了一个叫 runnext 的结构，本地队列和 runtext 都是为了解决程序的局部性，只有一个队列其实也是可以达到生产消费流程的，但是在计算机执行代码的过程中或者在大部分系统中，程序是有局部性原理的，也就是我们最近调用的代码非常有极大可能性得到再一次调用，这是一个非常有名的原理，它分为代码的局部性和数据的局部性，根据这里原理在我们右边刚刚创建的 goroutine 如果是非常高优先级的 goroutine,所以它会在本地的队列或者在runnext的结构中执行

优先级最高的是 runnext，其次是本地执行队列，再其次才是全局队列

上面提过多级队列就是为了解决程序的局部性问题，然后在解决程序局部性问题的时候尽量让生产和消费少加锁，如果只有一个队列的情况下其实也是可以实现这个流程的，这样的生产和消费就都需要加锁了

在这个过程中我们又不希望所有的生产都进全局队列，执行调度循环的不止是一个线程，而是多个线程并且是多个线程的消费，如果只有一个全局队列的话就需要所有的线程在消费这个全局队列的时候加锁，这是一个基本原则，所以就分为了本地队列和全局队列

当然在执行过程中有一部分线程可能是空闲的，这些空闲的线程，这些空闲的线程它会在一个专门的 midle 的队列里面进行管理，本身线程在执行过程中是按需创建的，所以总是会有一些线程是空闲下来，它们就会在队列里面等待下一次执行，就是没有可用线程的时候会到空闲线程里面找

goroutine 的生产端

代码起 routine 的时候会生成一个任务，左上角就是创建了一个 goroutine，goroutine 会调用 runtime 里面的 newproc 函数，这个函数会生成 goroutine 的结构，会指向具体的代码位置，一路执行下来，比如 malg && allgadd 就是在分配内存空间，并将它们管理在 allg 的数组结构里面，然后下面 runqput 就是再把他们放入到队列里面，右边就是入队的流程

因为队列是有优先级的，所以在生产 goroutine 的时候会优先把 goroutine 往当前绑定的 _p_ 的 runnext 中放，有没有放成功其实也是有区分的，_p_.runnext 本身也是有一个值的，也只有一个值，不是一个队列，可以理解为一个高优先级的队列，只是有一个元素

也就是当我将 goroutine 生产出来之后，它会先进 runnext，如果 runnext 原来有值，就需要好人做到底，送佛送到西，既然将别人踢出去了，就需要将别人先送到本地队列，如果本地队列也满了，就需要将刚才提出来的 G 送到全局队列里面，如果没有满就放到本地队列就可以了

上面的流程就是从生产 G 到进入队列的全流程，也即是上上面出现的 runnext，local run queue，global run queue 的过程

  • runnext 为空的情况

  1. 通过 runtime.newproc 生成一个 G；

  2. 生成的 G 要和三个队列进行交互，因为 runnext 为空，只需要将新生成的 G 通过 runtime.runqput 放进去就行了，到此一个生产流程就完成了，相当于生产代码执行完了；

  • runnext 非空的情况

  1. 通过 runtime.newproc 生成一个 G；

  2. 现在想要将新生成的 G 通过 runtime.runqput 放到 runnext 里面，但是现在 runnext 被一个 G 占着，就需要将老的 G 踢走，那需要将老的 G 提到什么地方呢？就需要将老的 G 提到 本地队列（local run queue）中，如果本地队列没有满，就可以成功将老的 G 从 runnext 成功踢出去，然后将新的 G 放到 runnext 中；

  • local run queue 满的情况

  1. 通过 runtime.newproc 生成一个 G；

  2. 现在想要将新生成的 G 通过 runtime.runqput 放到 runnext 里面，但是现在 runnext 被一个 G 占着，就需要将老的 G 踢走，需要将老的 G 提到 本地队列（local run queue）中

  3. 如果将老的 G 踢出去了，但是本地队列已经满了，就需要将本地队列里面一半的 goroutine 拿出来再加上刚刚从 runnext 踢出去的一个G 生成一个新的结构 batch

batch = len(local run local) / 2 + 1

  1. 将生成的 batch 拼成一个链表加入到全局队列里面 glocal run queue

本地队列是个数组，全局队列是个链表，为什么这样设计呢？

因为全局队列大小是无限的，但是本地队列只限制了 256 的大小，这其实也是由局部性原理决定的，本地访问肯定数组访问的比较快，因为数组内存是连续的，所以局部性会很好，理论上新进来的 goroutine 优先级要比老的高

goroutine 的消费端

消费端就是多个线程在反复执行一个循环，这个循环从队列中拿值，右边4个蓝色块表示一个标准的调度循环执行的流程，也是 runtime 里面的4个函数反复在执行，最重要就是 schedule 函数,
这个函数会从几个队列的源去找可以用的任务
