计算机是怎么执行我们的程序的呢？

CPU 无法理解文本，只能执行一条一条的二进制机器码指令
就是一行一行解析后面的二进制代码然后再去执行的，每次执行完一条指令，pc 寄存器 就指向下一条指令的位置继续执行，对于CPU来说没有什么负担，就挨个去解释每条指令并执行就可以了

从文本代码编译之后，最终生成文件里面只有汇编指令，有一个非常常见的通用寄存机叫 pc 寄存器，CPU就是靠 pc 寄存器 所指向的位置来执行我们的代码的，在 64 位平台上 pc 寄存器 = rip

runtime

  • 了解 Go 语言本身编译出来的二进制在预执行起来之后是怎么组成的?

GO 的可执行文件由两部分组成

  1. runtime

  2. 用户代码

Go 语言是一门有 runtime 的语言，可以认为 runtime 是为了实现额外的功能，由程序员提供的在程序运行时自动加载/运行的一些模块

上面是我们些的 GO 程序代码，下面就是 Runtime ，GO 代码通过一些比如 goroutine 的创建，或者 channel 的操作，或者内存的分配来和 Runtime 进行交互，Runtime 负责管理我们提交的一些任务，它和操作系统是通过系统调用和线程的调用来做交互的，也有人说 runtime 相当于 go 的 kernel ，可以这么理解

  • Go 语言的 runtime 包括：
  1. Scheduler

调度器管理所有的 G，M，P，在后台执行调度循环

  1. Netpoll

为了支持高并发，原生的在 runtime 里面实现了一套网络 FD 管理流程，比如网络轮询负责网络 FD 相关的读写，就绪事件

  1. Memory

当代码需要内存时，负责内存的分配工作

  1. Garbage

当内存不再需要时，负责内存回收

这些模块中（runtime 中的四座大山），最核心的就是 Scheduler 调度器，它负责串联所有的 runtime 流程

调度组件与调度循环

  • 每次写下上面的一段语句的时候，到底发生了什么？

其实就是向 runtime 提交了一个计算任务。func() { xxx } 里面包裹的代码就是计算任务的内容，它包含的一坨代码，这坨代码已经被编译到二进制文件里面了，每次启动一个 goroutine 就相当于向 runtime 提交了一个任务，既然有提交任务，就应该有人去消费这个任务，所以 Go 的调度流程本质上是一个 生产-消费 过程

平时开发中会写一些 生产-消费 程序，常见就是消息队列，在右边生产端会不停的提交任务，对于程序来说就是每次 go func 一次都是提交一个任务，这个任务肯定会进入某个队列，然后在队列的另一端会有人不停的消费这个任务去执行，执行完之后相当于这个任务被执行掉了，对于用户来说就是目标达到了，把我们的代码都执行了一遍；

在 runtime 中，这些都是有一些角色的，右边的生产者相当于每次 go func() {} 的时候都生产了一把，中间的队列（queue），任务会被打包成 协程 ，协程 在上面的模型叫 G， 也就是 goroutine ，goroutine 它最终是进入了专门的队列，队列的另一端消费的就是 线程-M，线程是在循环执行消费操；

Go 最重要的三个接口，G, M, P，上面的过程可以理解成 生产-消费 流程，右边不管什么地方起一个 goroutine，都相当于提交了一个计算任务，也就是起了一个 G，包装成一个 G，然后这个计算任务进入到一个队列，对端有人去消费，对端的线程一直执行循环去消费这些任务；

为了能够在性能上达到更好的目标，GO语言在队列上是有分级的，中间的队列会分为几个部分，上图中只列出了两个，但不是全部，比如有一个本地的执行队列 local run queue ，还有一个全局的执行队列 global run queue，对于 runtime 来说，它就是来协调这些对象之间的工作，并且进一步去做优化

go func 去了哪里？

右边黄色，蓝色，绿色部分就是线程，大部分线程（M）都是一直在执行调度循环，调度循环简（schedule loop）单理解就是从左边的各种队列里面把任务拿出来去执行，这就是线程的相关操作。

当然在执行过程中有一部分线程可能是空闲的，这些空闲的线程它会在一个专门的 midle 的队列里面去管理，本身线程在执行过程中是按需创建的，所以总是会有一些线程是空闲下来，它们就会在队列里面等待下一次执行，没有可用线程的时候会到空闲线程里面找；

左边的三个蓝色的就是任务来源，队列有一个本地队列 local run queue 和一个全局队列 global run queue，为了优化 GO 语言还专门做了一个叫 runnext 的结构，本地队列和 runtext 都是为了解决程序的局部性，只有一个队列其实也是可以达到生产消费流程的，但是在计算机执行代码的过程中或者在大部分系统中，程序是有局部性原理的，也就是我们最近调用的代码有极大可能性马上会再次得到调用，这是一个非常有名的原理，它分为代码的局部性和数据的局部性，所以我们右边刚刚创建的 goroutine 如果是非常高优先级的 goroutine,所以它会在本地的队列或者在runnext的结构中执行；

在这个过程中我们又不希望所有的生产都进全局队列，因为执行调度循环的不止是一个线程，而是多个线程并且是多个线程的消费，如果只有一个全局队列的话就需要所有的线程在消费这个全局队列的时候加锁，这是一个基本原则，所以就分为了本地队列和全局队列；

又因为代码和数据的局部性，说明刚刚创建的代码很可能是非常高优先级需要马上执行完的，所以在每个 P 上绑定了一个 runnext，为了管理这些 runnext 和 local run queue,专门做了 P 结构；

每个P都有一个runnext 和一个local run queue, 以及所有P共享一个global run queue

goroutine 的生产端

生产端也就是右边代码起 goroutine 的时候会生成一个任务，左上角就是创建了一个 goroutine， goroutine 会调用 runtime 里面的 newproc 函数，这个函数会生成 goroutine 的结构，会指向具体的代码位置，一路调用下来，比如 malg && allgadd 就是给 goroutine 分配内存空间，并将它们管理在 allgadd 的数组结构里面，然后下面 runqput 就是再把它们放入到队列里面，右边就是入队的流程了；

其实这个入队流程也没有那么简单，上面介绍了队列是分了三个级别的，优先级最高的是 runnext，其次是本地执行队列，再其次才是全局队列，因为队列是有优先级的，所以在生产 goroutine 的时候会优先把 goroutine 往当前绑定的 _p_.runnext 中放，有没有放成功其实也是有区分的，_p_.runnext 本身也是有一个值的,这个时候如果把别人踢掉了，就需要做一些善后工作,可以认为 P 是管理本地队列的，也只有一个值，不是一个队列，可以理解为一个高优先级的队列，只是有一个元素；

把 goroutine 生产出来之后，它会先进 runnext， runnext 如果原来有值，就需要好人做到底，送佛送到西，自己进去了，把别人踢出去了，既然将别人踢出去了，就需要将别人先送到本地队列，如果本地队列也满了，就需要将刚才提出来的 G 送到全局队列里面，如果没有满就放到本地队列就可以了;

上面的流程就是从生产 G 到进入队列的全流程，也即是上面出现的 runnext，local run queue，global run queue 的过程;

上面提过分多级队列，多级队列主要就是为了解决程序的局部性问题，然后在解决程序局部性问题的时候尽量让生产和消费流程少加锁，如果只有一个队列的情况下其实也是可以实现这个流程的，这样的生产和消费就都需要加锁了；

runnext 为空的情况

  1. 通过 runtime.newproc 生成一个 G；

  1. 生成的 G 要和三个队列进行交互，因为 runnext 为空，只需要将新生成的 G 通过 runtime.runqput 放进去就行了，到此一个生产流程就完成了，相当于生产代码执行完了；

runnext 非空的情况

  1. 通过 runtime.newproc 生成一个 G；

可以从下图中看到 runnext 原来是被一个 G 占着

  1. 现在想要将新生成的 G 通过 runtime.runqput 放到 runnext 里面，但是现在 runnext 被一个 G 占着，就需要将老的 G 踢走，那需要将老的 G 踢到什么地方呢？就需要将老的 G 踢到 本地队列（local run queue）中，如果本地队列没有满，就可以顺利将老的 G 从 runnext 成功踢出去，然后将新的 G 放到 runnext 中；

local run queue 满的情况

  1. 通过 runtime.newproc 生成一个 G；

  1. 上图中可以看到本地队列是满的，现在想要将新生成的 G 通过 runtime.runqput 放到 runnext 里面，但是现在 runnext 被一个 G 占着，就需要将老的 G 踢走，需要将老的 G 踢到 本地队列（local run queue）中

  1. 如果将老的 G 踢出去了，但是本地队列已经满了，就需要将本地队列里面一半的 goroutine 拿出来再加上刚刚从 runnext 踢出去的一个 G ，生成一个新的结构 batch，这个 batch 就是从本地队列里面拿了一半的 goroutine，再加上刚才新进来的也就是踢出去的 runnext

batch = len(local run local) / 2 + 1

  1. 将生成的 batch 拼成一个链表加入到全局队列里面 glocal run queue

本地队列其实是个数组，全局队列是个链表，为什么这样设计呢？

因为全局队列大小是无限的，但是本地队列只限制了 256 的大小，之所以本地队列使用数组其实也是由局部性原理决定的，本地访问肯定数组访问的比较快，因为数组内存是连续的，所以局部性会很好，理论上新进来的 goroutine 优先级要比老的高

goroutine 的消费端

消费端就是多个线程在反复执行一个循环，这个循环从队列中拿值，右边4个蓝色块表示一个标准的调度循环执行的流程，也是 runtime 里面的4个函数反复在执行，最重要就是 schedule 函数,
这个函数会从几个队列的源去找可以用的任务，图中红色部分都是和 gc 相关的逻辑

首先 schedule 会进行判断，也就是每执行 60 次本地队列的获取，就会去全局队列里面检查一次，这也是为了让全局队列里的那些东西也有机会能够被执行到，如果不去获取的话，全局队列里面的东西优先级就会低很多，就会对里面老的 goroutine 造成额外的延迟

每60次从全局队列里面获取一个 goroutine，获取到了就走后面的 runtime.execute 的流程，runtime.execute 执行的流程就是去执行 goroutine 的代码，后面还有 runtime.gogo 和 runtime.goexit 退出，然后再返回到调度流程里面；

左边的任意函数如果能返回一个可以执行的 goroutine 的话，调度循环就可以一直往后走；

如果全局队列中没有获取到，或者当前不需要执行全局，那就会去本地的 run queue 中去拿，本地 run queue 拿的话一般都是优先从 runnext 中拿，如果本地 runget 也没有获取到，就会执行下面的 findrunnable 函数，findrunnable 函数是会分为上半部分 top 和下半部分 stop;

top 部分也会反复的再去本地队列中拿，如果还是拿不到，就需要加锁去全局队列中获取，如果全局队列还是没有获取到，就需要去看下当前的网络轮询情况 netpoll，如果这个地方能找到 goroutine 的话，就直接拿去执行，并且把这些待执行的 goroutine 放到全局队列中去；如果 netpoll 执行完后还是没有拿到可以执行的 goroutine，这个时候就需要从其他的 P 里面去偷一部分过来，这个就是常说的 Work Stealing 的逻辑；

如果 top 部分执行完了还是没有获取，也就说明 M 没有机会得到执行了，下面也就会进入休眠流程了，在休眠之前还会锤死挣扎一下，再去从全局队列获取一下，然后再检查一遍所有的 run queue，再执行一次 netpoll，如果还是获取不到，就将当前的 M 停掉，这就是一个比较完整的消费端流程

什么是 M:N 模型

为什么说 go 调度器是个 M:N 模型？

M: 虽然 go 里面做了一些抽象，比如抽象出来的 goroutine, 可能很多人认为 goroutine 就是执行实体，其他 goroutine 不是执行实体，它只是一个计算任务，真正的执行实体不管什么语言最终都是线程，线程可以认为是 Linux 最小的调度单元，对于 go 来说也是如此，所有的逻辑运转和代码执行最终都是由 M 来做的，M 才是整个流转流程的核心，也是和系统里面的线程一一对应的

P: 可以认为是 M 要执行任务时必须要获取到的 Token，必须要要有这个 Token，才会在 CPU 上有执行的权利，但是大部分人认为它是一个虚拟处理器

阻塞

实际上在我们的代码里面有一些是需要和操作系统交互的，操作系统交互是通过系统调用，但是这些系统调用有可能会阻塞，除了系统调用，还会其他各种阻塞，如果这些阻塞在 go runtime 里面不去处理就会出现很多问题

比如 M 绑定了 P才能去执行，现在有 8 个 P，如果有 8 个线程正在执行阻塞操作，比如在操作 channel, 可能卡了，这个时候没有任务的 P，其他新来的 goroutine 创建出来的线程没有办法获取到 P，此时整个系统的吞吐量就下去了，所以 runtime 就需要能处理用户的这些阻塞任务

阻塞的几种场景

在线程发生阻塞的时候会无限制地创建线程么？

并不会！！！

第一个场景 make 了一个 buff 是 0 的 channel，直接往里面塞数据，这个时候其实是会阻塞的
第二个场景 make 了一个 buff 是 0 的 channel，从里面消费数据，这个时候也是会阻塞的

执行 time.Sleep() 用户态的代码也是会阻塞的

这种如果执行网络的读写，比如执行网络的读，如果当前网络连接里面没有数据可以读，这时候代码也是会阻塞的

这种是网络的写

这种是执行锁操作，如果锁已经被别人占有了，此时调用 Lock() 也是会被阻塞的

上面的 6 中情况都是会被 runtime 处理掉的，也就是说这 6 种阻塞都不会阻塞调度循环，它的底层一定会让 goroutine 进入某一个等待的结构体，然后它在当前的线程在继续执行 Schedule 函数

下面列举了上面 6 种情况在 runtime 里面处理的时候是怎么塞到等待的结构里面的

上面是 channel 发送的情况，channel 发送如果阻塞了，在 channel 上会有个等待队列，也就是图中的 sendq，有 first 和 last, 其实就是个简单的链表，first 和 last 最终会把 goroutine 打包成 sudog 的内置的数据结构塞到等待队列上

接收的阻塞和发送的阻塞差不多，最终也是在一个等待队列上

上面是连接的写阻塞和连接的读阻塞，读写的 goroutine 最终会被挂在底层的数据结构上，比如 wg 就是 write goroutine，rg 就是 read goroutine

上面的 select 的阻塞，chan1 ,chan2 ,chan3 分别是读，发，读的操作，说明当前 select 是无法继续执行了，当前的 goroutine 是会挂在三个结构上，比如 chan1 的读取队列，当前 goroutine 挂在尾巴上，chan2 发送队列也是挂在尾巴上，chan3 收取队列也是挂在尾巴上，所以一个 goroutine 会挂载三个 sudog 上

time.Sleep() 会把结构挂在 timer 结构的一个参数上

上图是应用阻塞在锁上的情况，如果获取不到锁的话，会把当前 goroutine 也会最终打包成 sudog，然后这个 sudog 会停留在 数堆 的结构上，数堆 是个二叉平衡树，这个二叉平衡树的每个节点都是一个链表

上面列出的 6 种情况在 runtime 里面都有处理，只要 runtime 有处理的这些情况，就不会将线程阻塞住，然后线程就可以不断的去执行其他任务

  • 为啥有的等待的是 sudog?有的是 g?

上图是 sudog 在 runtime 中的一段注释。涉及到同步的问题，比如之前select , 会阻塞在三个 channel 上，三个 channel 也就以为着同一个 goroutine 对应着三个 sudog

g 和 sudog 对应的一对多的关系，所以在涉及到同步操作的时候，把 goroutine 重新封装成 sudog，如果不涉及到同步，直接把 g 挂到等待列表上就可以了

还有一些是 runtime 无法拦截的，一种是 CGO，还有一种是和系统交互的 syscall，既然无法处理，就说明必须要在阻塞的时候会堵住一个线程，比如8核心的情况下，有8个线程正在执行 syscall，就需要把这些阻塞在 CGO 或者 syscall 的线程的 P 结构剥离掉

这需要后台有个专门的后台的 sysmon 的线程来管理这些工作，sysmon 是 system monitor 的缩写，它在 runtime 里面是个高优先级的线程和 goroutine，它主要的工作有三个

第一个就是检查当前所有线程是不是都已经阻塞住了，这个时候就说明程序写的有问题，就会崩溃来直接提示；
第二个是网络轮询器，它会在 sysmon 执行一次，netpoll 为了保证它的高优先级，这段代码到处可见；
第三个是 retake ，没有办法处理的阻塞，这个时候需要把线程上的 P 和 M 的绑定剥离开，另外一个是新加的逻辑，就是如果用户 g 运行很久，一般情况下是 10 ms，就需要在 sysmon 里发一个信号，让用户执行的 G 停下来，让 G 进入队列继续等待

剥离开后会把 G 放入到全局队列，system monitor 是个循环执行的流程，它会自动执行，右上角是每执行完一次流程后需要等待的时间，如果当前在执行 system monitor 的时候抢占了一次就会将 10ms 重新调回到 20us 中去 ，system monitor 的高优先级体现在它不需要 P 就可以执行
