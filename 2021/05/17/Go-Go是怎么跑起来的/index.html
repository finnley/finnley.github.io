Go 程序 hell.go 的编译过程

  1. 文本
  1. 编译

单个文件可以通过 go build xxx.go 生成可执行文件，这就是简单的代码编译过程，从文本编译到二进制文件，其实这个编译过程也是有很多阶段的，这些属于编译原理内容（有时间可以研究）

可以通过 go build -x hello.go 命令在编译的时候可以将详细内容输出出来

编译：文本代码 -> 目标文件（.o, .a）
链接：将目标文件合并为可执行文件

编译阶段就是将文本代码转成目标文件，这个目标文件需要和标准库里已经编译出来的静态库做链接，链接了之后就会在临时目录里面把编译后完整的二进制文件输出出来，它再把临时目录中的文件挪动到默认编译出来 -o 的目标，然后再将临时目录删掉，这就完成了整个编译过程

可执行文件

可执行文件在不同操作系统上规范不一样

Linux Windows MacOS
ELF PE Mach-O

Linux 的可执行文件 ELF (Executable and Linkable Format) 为例，ELF 由几部分构成：

  • ELF header

  • Section header

  • Sections

如果想要找 go 语言的入口，只需要看 ELF header 就可以了

操作系统执行可执行文件的步骤（以 Linux 为例）

  1. 解析 ELF Header

  2. 加载文件内容至内存

  3. 从 entry point 开始执行代码

操作系统加载可执行文件，需要解析 ELF Header，通过 ELF Header 里面的内容找到相应的段，然后再将二进制文件的段加载到内存，然后再通过 ELF Header 中的 entry point 开始执行我们的代码

计算机是怎么执行我们的程序的呢？

目标：了解计算机CPU是怎么执行二进制文件中的指令

从文本代码编译之后，最终生成文件里面只有汇编指令，有一个非常常见的通用寄存机叫 pc 寄存器，CPU就是靠 pc 寄存器 所指向的位置来执行我们的代码的，在 64 位平台上 pc 寄存器 = rip

CPU 无法理解文本，只能执行一条一条的二进制机器码指令
就是一行一行解析后面的二进制代码然后再去执行的，每次执行完一条指令，pc 寄存器就指向下一条继续执行，对于CPU来说没有什么负担，就挨个去解释每条指令并执行就行了

runtime

目标：了解 Go 语言本身编译出来的二进制在预执行起来之后是怎么组成的

GO 的可执行文件由两部分组成

  1. runtime

  2. 用户代码

Go 语言是一门有 runtime 的语言，可以认为 runtime 是为了实现额外的功能，有程序员提供的在程序运行时自动加载/运行的一些模块

上面是我们些的 GO 程序代码，下面就是 Runtime ，GO 代码通过一些比如 goroutine 的创建，或者 channel 的操作，或者内存的分配来和 Runtime 进行交互，Runtime 负责管理我们提交的一些任务，它和操作系统是通过系统调用和线程的调用来做交互的，也有人说 runtime 相当于 go 的 kernel ，可以这么理解

  • Go 语言的 runtime 包括：
  1. Scheduler

调度器管理所有的 G，M，P，在后台执行调度循环

  1. Netpoll

为了支持高并发，原生的在 runtime 里面实现了一套网络 FD 管理流程

  1. Memory

当代码需要内存时，负责内存的分配工作

  1. Garbage

当内存不再需要时，负责内存回收

这些模块中（runtime 中的四座大山），最核心的就是 Scheduler 调度器，它负责串联所有的 runtime 流程
